## Interview & Demo Reference

## 1. Project Summary

**Promotion Analytics AI Assistant** is a production-grade agentic business analytics system that converts natural-language questions into validated SQL queries, executes them against a DuckDB analytical database, and returns grounded, business-friendly insights through a Streamlit chat interface.

Built across 12 phases, the system demonstrates end-to-end AI engineering: from synthetic data generation and vector-store indexing, through multi-agent orchestration and SQL validation, to real-time monitoring and observability.

**Stack:** Python · Groq (LLaMA-3.3-70B) · LangChain · DuckDB · ChromaDB · Streamlit · Pydantic

## 3. One Key Challenge Faced

**Challenge:** Ensuring the LLM-generated SQL only accessed semantic-layer views and never raw warehouse tables (e.g., `sales_raw`, `inventory_raw`).

**Why it matters:** Raw table access bypasses business logic embedded in views (e.g., date filters, join conditions), risking incorrect or misleading answers.

**Solution implemented:**
1. **Whitelist enforcement** in `QueryGenerationAgent` — a static set of approved view names (`vw_weekly_sales`, `vw_weekly_inventory`, etc.) is checked against every generated SQL statement using regex extraction.
2. **View-first system prompt** — the LLM prompt explicitly lists only approved views and their schemas, so the model has no schema information for raw tables.
3. **Validator double-check** — `SQLValidator` re-inspects the SQL before execution as a second safety layer.
4. **Trailing semicolon fix** — DuckDB's `COUNT(*) FROM (sql) t` wrapper pattern fails when the inner SQL ends with `;`. Fixed by stripping trailing semicolons before wrapping.

---

## 4. One Key Limitation

**Current limitation:** The system uses a synthetic dataset (generated by Faker) and a hand-crafted semantic layer. All business metric definitions (`lift`, `effectiveness`, `reduction`) are fixed in a JSON glossary.

**Impact:** The system cannot self-adapt to a new domain (e.g., healthcare, e-commerce) without engineering intervention to:
- Update `rag/glossary.json` with domain-specific metric definitions
- Regenerate the ChromaDB vector index
- Update `db/semantic_layer.sql` with domain-specific views
- Add few-shot examples to `rag/few_shot_bank.json`

**Future improvements:**
- Support real enterprise data warehouses (Snowflake, BigQuery, Redshift)
- Auto-discover schemas and generate views from table metadata
- Fine-tune the LLM on domain-specific SQL patterns
- Add a self-healing loop that automatically updates the glossary from analyst feedback

## 5. Interview Questions I Should Be Ready For

### "Why not pure RAG? Why generate SQL?"

Pure RAG retrieves pre-existing text chunks and synthesises an answer from them. It cannot answer **aggregated, filtered, or computed questions** like "What was the week-over-week revenue change in South region?" because the answer doesn't exist in any pre-written document — it must be **computed** at query time from live data.

SQL generation allows:
- Arbitrary aggregations (`SUM`, `AVG`, `GROUP BY`)
- Time-window filtering (`WHERE week BETWEEN X AND Y`)
- Cross-dimensional comparisons (region × category × week)
- Exact numeric answers, not probabilistic paraphrases

**RAG is still used** in this project — but for **query grounding** (resolving business term definitions), not for answer generation.

---

### "What is query grounding?"

Query grounding is the process of resolving ambiguous business language to **precise, unambiguous metric definitions** before SQL generation.

For example: the phrase *"improve sales"* could mean:
- Total revenue increased
- Units sold increased
- Revenue minus baseline cost increased
- Week-over-week delta is positive

Without grounding, the LLM must guess. With grounding, `QueryGroundingAgent` retrieves the canonical definition of `"effectiveness"` from ChromaDB and injects it into the SQL generation prompt:

```
metric: effectiveness
formula: (promo_sales - baseline_sales) / baseline_sales * 100
comparison: week-over-week delta
```

This makes every query **deterministic and auditable**.

---

### "What is a semantic layer?"

A semantic layer is a set of pre-defined SQL views that expose curated, business-friendly data abstractions over raw tables. It encodes business logic (joins, filters, aggregations) into views so that:

1. The LLM only needs to know view names and column names — not raw schema complexity
2. Business rules (e.g., "week 1–52 fiscal calendar") are enforced once, centrally
3. SQL generation surface area is minimised, reducing hallucination risk

In this project, the semantic layer is defined in `db/semantic_layer.sql`:
```sql
CREATE VIEW vw_weekly_sales AS ...
CREATE VIEW vw_weekly_inventory AS ...
CREATE VIEW vw_promo_summary AS ...
```

---

### "How do you prevent hallucinations?"

Multi-layer defence:
1. **Grounding** — metric definitions are retrieved from a curated vector store, not invented by the LLM
2. **View whitelist** — SQL can only reference approved semantic-layer views
3. **Schema injection** — only whitelisted column names are given to the LLM in the prompt
4. **Validator** — syntax and row-count checks catch nonsensical queries before execution
5. **Fallback responses** — if validation fails twice, the system returns a structured clarification request rather than a fabricated answer
6. **Coverage flag** — the `ResponseSynthesizer` detects partial data and warns the user

---

### "How does validation work?"

`SQLValidator` runs two checks in sequence:

1. **Syntax check** — `DuckDB.execute("EXPLAIN " + sql)` — if DuckDB's query planner rejects the SQL, validation fails immediately
2. **Row-count estimation** — wraps the SQL in `SELECT COUNT(*) FROM (sql) t` — if the estimated result is 0 rows or exceeds the `ROW_COUNT_MAX` bound, it triggers regeneration

On failure, a `RegenerationSignal` is returned with the error message injected back into the next SQL generation prompt: `"PREVIOUS ATTEMPT FAILED WITH ERROR: {reason}. Please fix the SQL."`

---

### "Why use DuckDB?"

DuckDB is an **in-process analytical SQL engine** optimised for:
- OLAP-style queries (aggregations, GROUP BY, window functions)
- Reading directly from pandas DataFrames and CSV files
- Zero infrastructure overhead — runs entirely in-process, no server needed
- Fast startup — suitable for Streamlit-embedded analytics

It is ideal for prototyping and demonstrating analytics pipelines without requiring a cloud data warehouse.

---

### "What is the role of ChromaDB?"

ChromaDB is used as the **vector store for the RAG retrieval layer** in `QueryGroundingAgent`.

It stores:
1. **Business metric definitions** from `rag/glossary.json` (e.g., `effectiveness`, `lift`, `reduction`, `baseline`)
2. **Few-shot SQL examples** from `rag/few_shot_bank.json` (example question→SQL pairs per topic)

At query time, the user question is embedded using `sentence-transformers/all-MiniLM-L6-v2` and compared against stored embeddings. The top-k most similar definitions and examples are retrieved and injected into the SQL generation prompt.

This grounds the LLM in domain-specific business language, ensuring that `"improve sales"` always resolves to the canonical `effectiveness` formula.

---

### "How does session memory work?"

The `SessionMemory` class in the orchestrator stores per-session state:
- `last_question` — the previous user question
- `last_intent` — the classified `Intent` object (topic, region, category, SKU)
- `last_sql` — the previously generated SQL
- `last_response` — the serialised response dict

For multi-turn queries, the orchestrator prepends a context string to the current question:
```
"Previous context: topic=region_comparison, region=South, category=None.
Current Question: What about North?"
```

This allows the intent classifier to resolve ambiguous follow-ups using prior context.

## 6. Key Design Decisions

| Decision | Rationale |
|---|---|
| Separate grounding from SQL generation | Keeps metric definitions stable; SQL agent focuses only on structure |
| View whitelist before execution | Prevents raw table access and prompt injection via table names |
| Pydantic for all agent outputs | Guarantees structured, type-safe contracts between agents |
| Never-crash guarantee | All exceptions return a structured `SynthesizedResponse` fallback |
| CSV-backed metrics logging | Zero infrastructure; readable by pandas for instant dashboard visualisation |
| Row-count bounds validation | Prevents empty results (0 rows) and pathological cartesian joins (> 500 rows) |
